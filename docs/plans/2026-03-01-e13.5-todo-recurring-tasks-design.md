# E13.5 — Séparation Todo / Tâches récurrentes + Dashboard amélioré

> **IMPORTANT** : Ce plan est un enrichissement POST-E13. Ne pas commencer avant que E13 soit terminé.
> E13 doit être complètement fini et commité avant de toucher à E13.5.

## Contexte

E13 backend implémente le CRON dynamique, les champs SQLite (cron_expression, cron_enabled, cron_prompt),
les API endpoints et le subagent tasks avec support CRON. Le dashboard affiche les tâches récurrentes
dans le Kanban avec des badges 🔄/⏸️, mais tout est mélangé sur une seule page.

Inspiré par le projet Gravity Claw (agent personnel Telegram TypeScript), on enrichit le dashboard
pour séparer clairement les tâches manuelles (Todo/Kanban) des automations récurrentes (Tâches),
et on prépare le terrain pour le cost tracking (E14).

## Objectifs

1. **Renommer** la page actuelle "Tâches" → "Todo" (Kanban pour tâches manuelles uniquement)
2. **Créer** une nouvelle page "Tâches" dédiée aux automations récurrentes
3. **Tracer les exécutions** : table task_executions avec durée, tokens, coût, résultat
4. **Préparer E14** : champs model/cost_estimate posés mais vides pour l'instant

## Étapes d'implémentation

### Étape 1 : Renommage Sidebar + déplacement Kanban
- Créer `app/todo/page.tsx` (copier contenu actuel de tasks/page.tsx)
- Filtrer les tâches récurrentes du Kanban (ne garder que cron_enabled !== 1)
- Mettre à jour sidebar.tsx avec les nouveaux liens/labels
- Remplacer `app/tasks/page.tsx` par un placeholder pour l'étape 4

### Étape 2 : Table task_executions (SQLite)
```sql
CREATE TABLE IF NOT EXISTS task_executions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  task_id TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'success',  -- 'success' | 'error'
  duration_ms INTEGER,
  tokens_in INTEGER,
  tokens_out INTEGER,
  model TEXT,
  cost_estimate REAL,           -- coût estimé en USD (préparation E14)
  result_summary TEXT,          -- résumé court du résultat
  error_message TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```
- Migration dans `initSchema()` via table `_migrations`
- Fonctions : `logTaskExecution()`, `listTaskExecutions(taskId, limit)`
- Modifier `cron.ts` → wrapper l'exécution pour capturer durée, tokens, résultat

### Étape 3 : API endpoints exécutions
- Enrichir `GET /api/tasks/recurring` avec stats (last_run, next_run, success_rate, total_cost)
- `GET /api/tasks/:id/executions?limit=20` — historique d'exécutions
- `POST /api/tasks/:id/execute` — exécution manuelle immédiate
- Ajouter dépendance `cron-parser` pour calculer next_run

### Étape 4 : Page Tâches récurrentes (dashboard)
Vue en **liste/tableau** (pas Kanban) :

| Colonne | Source |
|---------|--------|
| Nom | task.title |
| Fréquence | humanCron(task.cron_expression) — déjà implémenté dans task-card.tsx |
| Statut | Actif 🟢 / Pausé ⏸️ / Erreur 🔴 |
| Prochaine exécution | calculé depuis cron_expression (cron-parser) |
| Dernière exécution | depuis task_executions |
| Modèle | task_executions.model (ou "default" pour l'instant) |
| Coût ce mois | SUM(cost_estimate) du mois |
| Actions | ▶️ Exécuter / ⏸️ Pause / ✏️ Éditer / 🗑️ Supprimer |

### Étape 5 : Panneau détail tâche récurrente
Panneau slide-in (même pattern que TaskDetailPanel existant) :
- **En-tête** : titre éditable, badge statut
- **Configuration** : cron_expression (éditable), cron_prompt (éditable), modèle (dropdown futur E14)
- **Stats** : exécutions ce mois, taux de succès, coût total, coût moyen/exécution
- **Timeline** : historique des exécutions (date, durée, statut, coût, résumé résultat)
- **Actions** : Exécuter maintenant, Activer/Désactiver, Supprimer
